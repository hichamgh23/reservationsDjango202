1. Méthodologie et techniques de développement

Le projet a été développé de manière itérative, fonctionnalité par fonctionnalité, en suivant le plan de cours chapitre par chapitre. Chaque itération livrait un ensemble cohérent et fonctionnel versionné sur Git avant de passer à la suivante.

Architecture

Pattern MVT (Model – View – Template) imposé par Django :

  • Model — 13 modèles ORM mappés sur MariaDB (catalogue/models.py)
  • View — fonctions et une classe Python (catalogue/views.py)
  • Template — moteur Django avec héritage via base.html

Principes appliqués

  • Séparation stricte HTML / CSS / JS — un fichier externe par composant
  • DRY via {% extends %} et {% include %} dans les templates
  • Contrôle d'accès déclaratif : @login_required et décorateur @group_required personnalisé
  • API REST exposée via Django REST Framework
  • Données exportables en CSV et en flux RSS

7. Gestion de la sécurité

Protection CSRF

Le middleware CsrfViewMiddleware est actif. Tous les formulaires POST incluent le token {% csrf_token %}. La vue de déconnexion utilise @csrf_exempt uniquement car elle accepte un POST direct sans formulaire Django standard.

Authentification

  • Système Django natif (django.contrib.auth) — mots de passe hashés PBKDF2-SHA256
  • Sessions sécurisées via django.contrib.sessions
  • Décorateur @login_required sur toutes les vues nécessitant une connexion
  • Validation : longueur minimale 6 caractères, vérificateur de mots de passe communs

Autorisations par profil

ProfilCondition techniqueAccès autorisé
Public Utilisateur non connecté Lecture catalogue, spectacles, artistes, lieux, localités, flux RSS
MEMBER Utilisateur authentifié (groupe MEMBER) Public + réservations + profil + dépôt d'avis + API REST
Superuser / Staff is_superuser = True Tout + CRUD artistes / localités + export CSV + Django Admin

Le décorateur @group_required('ADMIN') défini dans views.py autorise les superusers automatiquement, même sans appartenance explicite au groupe ADMIN.

Intégrité de la base de données

Toutes les clés étrangères définissent explicitement ON DELETE et ON UPDATE au niveau SQL (CASCADE, SET NULL ou RESTRICT), en complément du comportement ORM Django.

Prérequis à installer sur une nouvelle machine

  1. Python 3.14+ — https://www.python.org/downloads/ — cocher "Add to PATH"
  2. MariaDB 10.6 — https://mariadb.org/download/ — installer comme service Windows, noter le mot de passe root défini pendant l'installation
  3. HeidiSQL — https://www.heidisql.com/ — client graphique pour gérer MariaDB
  4. Git — https://git-scm.com/

Installation pas à pas

Étape 1 — Récupérer le projet depuis GitHub

git clone https://github.com/hichamgh23/reservationsDjango202.git
cd reservationsDjango202

Étape 2 — Créer et activer l'environnement virtuel Python

python -m venv venv

# Windows PowerShell
venv\Scripts\Activate.ps1

# Windows CMD
venv\Scripts\activate.bat

Étape 3 — Installer les dépendances Python

pip install -r requirements.txt

Étape 4 — Créer la base de données avec HeidiSQL

  1. Ouvrir HeidiSQL → Gestionnaire de sessions → Nouveau
  2. Paramètres de connexion :
    Type : MySQL (TCP/IP) | Hôte : 127.0.0.1 | Port : 3306
    Utilisateur : root | Mot de passe : celui défini à l'installation de MariaDB
  3. Cliquer Ouvrir, puis dans l'onglet Requête, exécuter :
CREATE DATABASE reservations CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'rooot'@'127.0.0.1' IDENTIFIED BY 'rooot';
GRANT ALL PRIVILEGES ON reservations.* TO 'rooot'@'127.0.0.1';
FLUSH PRIVILEGES;

Étape 5 — Importer le dump SQL (structure + données)

  1. Dans HeidiSQL, sélectionner la base reservations dans le panneau gauche
  2. Menu Outils → Importer un fichier SQL…
  3. Sélectionner le fichier dump_reservations.sql fourni à la racine du projet
  4. Cliquer Ouvrir — toutes les tables et données sont créées automatiquement
Le dump inclut : structure complète + FK + CHECK + données de démonstration (7 spectacles, 12 artistes, 9 localités, 7 lieux, 28 représentations, 7 réservations, 9 avis, 3 utilisateurs).

Étape 6 — Vérifier settings.py

Le fichier reservations/settings.py est déjà configuré :

DATABASES = {
    'default': {
        'ENGINE':   'django.db.backends.mysql',
        'NAME':     'reservations',
        'USER':     'rooot',
        'PASSWORD': 'rooot',
        'HOST':     '127.0.0.1',
        'PORT':     '3306',
        'OPTIONS':  {'charset': 'utf8mb4'},
    }
}

Étape 7 — Lancer le serveur

python manage.py runserver
